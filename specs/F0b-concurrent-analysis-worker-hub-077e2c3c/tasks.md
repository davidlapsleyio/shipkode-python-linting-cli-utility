# Implementation Plan: Concurrent Analysis Worker Hub




## Overview




The implementation follows a domain-driven approach, starting with the core coordination and aggregation logic before moving to the infrastructure-heavy parallel execution layer. This ensures that the logic for combining results is independent of whether those results were generated by processes, threads, or a single loop, facilitating the 'Result Consistency Invariant'. 

The second phase introduces the WorkerHub adapter using the 'multiprocessing' capabilities (or equivalent worker threads in the target environment) to handle the distribution logic and resource management. This phase specifically addresses container-awareness to satisfy requirement E25. Finally, we implement property-based tests to rigorously verify that parallel execution and single-threaded execution yield identical results across varying workloads.

- [ ] F0b-T1.3 Checkpoint: Domain Logic Verification

- Write unit tests for `ResultAggregator` to ensure merging two sets of results preserves all error counts.

- Mock `IWorkerBridge` to verify `AnalysisCoordinator` correctly partitions 50 projects into the expected number of chunks.

- _Requirements: E10_



- [ ] F0b-T2. Adapter Layer: Worker Hub and Resource Scaling

- [ ] F0b-T2.1 Implement Resource-Aware WorkerHub Architecture

- Create `WorkerHub` in `src/adapters/worker_hub.ts`.

- Implement CPU core detection logic.

- Add logic to read from cgroups or environment variables (e.g., CI_CORES) to support container-aware scaling.

- _Requirements: E25_


- [ ] F0b-T2.2 Implement Concurrent Execution and Error Handling

- Implement the worker execution loop (Process/Thread pool).

- Wrap file analysis calls in try-catch blocks within the worker.

- Return a 'PartialFailure' object if an individual file fails, ensuring the worker continues to the next file in its queue.

- _Requirements: E8_


- [ ] F0b-T2.3 Checkpoint: Integration Testing

- Integrate `WorkerHub` with `AnalysisCoordinator` via the `IWorkerBridge`.

- Verify that the full chain from CLI -> Coordinator -> WorkerHub -> Aggregator works for a small set of files.

- _Requirements: E8, E10_



- [ ] F0b-T3. Verification and Scaling Maintenance

- [ ] F0b-T3.1 Property Test: Execution Parity

- Create property-based tests (using a library like fast-check or hypothesis).

- Test Input: Random sets of mock files with various linting outcomes.

- [ ] F0b-T3.2 Scale Optimization: Incremental Processing *

- Implement incremental processing logic where only changed files are sent to `WorkerHub`.

## Notes




- Tasks marked with * (e.g., 3.2) are optional for MVP but recommended for production-grade CI environments.

- Requirement references (e.g., E8) ensure every architectural decision is traced back to user needs.

- Ordering constraint: The Aggregate use case must be finalized before the WorkerHub adapter to ensure the adapter satisfies the interface contracted by the domain.

- Backward compatibility: The 'sequential' mode is maintained as a fallback and as the baseline for the Result Consistency Invariant.
